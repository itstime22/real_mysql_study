# 트랜잭션과 잠금

**트랜잭션: 작업의 완전성(정합성) 보장 → 논리적인 작업 셋을 모두 완벽히 처리 or 원상태로 복구**

**잠금(lock): 동시성을 제어 → 한 시점에는 하나의 커넥션만 변경할 수 있게 해줌**

# 1. 트랜잭션

**트랜잭션을 지원하지 않는 MyIsam**과 **트랜잭션 지원하는 InnoDB**의 처리 방식 비교

## 📌 MySQL에서의 트랜잭션

- **InnoDB vs MyBatis - 트랜잭션 관점의 차이**
    - 예시
    1. 테스트용 테이블에 레코드 저장
        
        ```java
        mysql> CREATE TABLE tab_myisam (fdpk INT NOT NULL, PRIMARY KEY (fdpk)) ENGINE=MyISAM;
        mysql> CREATE TABLE tab_innodb (fdpk INT NOT NULL, PRIMARY KEY (fdpk)) ENGINE=INNODB;
        
        mysql> INSERT INTO tab_myisam (fdpk) VALUES (3);
        mysql> INSERT INTO tab_innodb (fdpk) VALUES (3);
        ```
        
    2. AUTO-COMMIT 모드에서 쿼리문 각각 실행
        
        ```java
        mysql> SET autocommit=ON;
        
        mysql> INSERT INTO tab_myisam (fdpk) VALUES (1), (2), (3);
        mysql> INSERT INTO tab_innodb (fdpk) VALUES (1), (2), (3);
        ```
        
    3. 테스트 결과 (둘다 프라이머리 키 중복 오류로 실패 - 이미 3이 있음)
        - **MyIsam**: 1, 2, 3 출력 → **이미 INSERT된 1,2를 그대로 두고 종료 (부분 업데이트) → 정합성 X**
        - **innodb**: 3 출력 → **오류 발생했기에 트랜잭션 원칙대로 쿼리문 실행 전으로 복구**
        
    - 결론: **트랜잭션**은 실패한 쿼리 결과에 대한 재처리 작업을 없애주는 **DBMS의 필수적 기능**
    

## 📌 트랜잭션 주의사항

- **트랜잭션은 필요한 최소의 코드에만 적용 (범위 최소화)**
- **서버 처리 예시** (게시물 작성 후 저장 버튼 클릭)
    1. 처리 시작
     -> 데이터베이스 커넥션 생성
     -> **트랜잭션 시작**
    2. 사용자의 로그인 여부 확인
    3. 사용자의 글쓰기 내용의 오류 여부 확인
    4. 첨부로 업로드된 파일 확인 및 저장
    5. 사용자의 입력 내용을 DBMS에 저장
    6. 첨부파일 정보를 DBMS에 저장
    7. 저장된 내용 또는 기타 정보를 DBMS에서 조회
    8. 게시물 등록에 대한 알림 메일 발송
    9. 저장된 내용 또는 기타 정보를 DBMS에 저장
      <- **트랜잭션 종료 (COMMIT)**
      <- 데이터베이스 커텍션 종료
    10. 처리 완료
    
    - 문제점
        - 실제 DBMS 데이터 저장 작업 → **5번부터 시작 (2,3,4번은 트랜잭션 포함 필요 X)**
        - 8번 작업의 **전송이나 원격 서버와 통신 작업은 DBMS의 트랜잭션에서 반드시 제거**해야함.
            
            → 프로그램 실행하는 동안, 원격 서버와 통신 못하는 경우 서버 위험
            
        - 현재 서버 처리 예시에는 DBMS 작업이 4개있음. **5,6번 작업은 반드시 하나의 트랜잭션으로 묶어야**하며 7번은 단순 조회이므로 트랜잭션에 포함 필요 X

- **서버 처리 예시 재설계 (기존 문제점 보완)**
    1. 처리 시작
    2. 사용자의 로그인 여부 확인
    3. 사용자의 글쓰기 내용의 오류 여부 확인
    4. 첨부로 업로드된 파일 확인 및 저장
     -> **데이터베이스 커넥션 생성** (또는 커넥션 풀에서 가져오기)
     -> **트랜잭션 시작**
    5. 사용자의 입력 내용을 DBMS에 저장 → **필요한 부분부터 트랜잭션 시작 / 5,6번 묶기**
    6. 첨부파일 정보를 DBMS에 저장
     <- **트랜잭션 종료** (COMMIT)
    7. 저장된 내용 또는 기타 정보를 DBMS에서 조회
    8. 게시물 등록에 대한 알림 메일 발송 → **메일 전송 작업은 트랜잭션에서 제거**
     -> **트랜잭션 시작**
    9. 저장된 내용 또는 기타 정보를 DBMS에 저장
     <- **트랜잭션 종료** (COMMIT)
     <- **데이터베이스 커넥션 종료**  (또는 커넥션 풀에 반납)
    10. 처리 완료

- 보완점
    - **커넥션 범위와 트랜잭션이 활성화돼 있는 프로그램의 범위 최소화**
    - **네트워크 작업은 트랜잭션에서 배제**

# 2. MySQL 엔진의 잠금

**MySQL의 잠금 = 스토리지 엔진 레벨 + MySQL 엔진 레벨**

**MySQL 엔진 레벨 잠금은 모든 스토리지 엔진에 영향**

1. **테이블 락** → 테이블 데이터 동기화 목적
2. **메타데이터 락** → 테이블의 구조 잠금
3. **네임드 락** → 사용자의 필요에 맞게 사용

**스토리지 엔진 레벨 잠금은 스토리지 엔진 간 상호영향 X**

## 📌 글로벌 락

- `FLUSH TABLES WITH READ LOCK` 명령어로 획득 가능
- MySQL의 잠금 중 가장 범위가 큼 (**MySQL 서버 전체에 영향**)
- **한 세션 글로벌 락**을 획득 → **다른 세션에서 SELECT을 제외한 DDL, DML 문장이 대기 상태**가 됨
- 가급적 **사용하지 않는 것이 좋음** (모든 테이블에 큰 영향을 주기에)
- MySQL 8.0 이후, InnoDB엔진이 기본 엔진으로 채택됐기에 **가벼운 글로벌 락 필요 → 백업 락 도입**

## 📌 테이블 락

- **테이블 락: 개별 테이블 단위로 설정되는 잠금**
- 명시적 or 묵시적으로 특정 테이블 락 획득
    - **명시적** → `LOCKA TABLES table_name [READ | WRITE]` 명령어
        - **글로벌 락 처럼 큰 영향을 주기에 사용 지양**
    - **묵시적** → MyISAM, MEMORY 테이블에 **데이터 변경 쿼리 실행 시 발생**
        - MySQL 서버가 데이터가 변경되는 테이블에 **잠금 설정 → 데이터 변경 → 즉시 잠금 해제** 
        (자동 획득, 자동 해제)
        - **InnoDB 테이블**은 단순 데이터 변경 쿼리로 인해 묵시적 테이블 락 설정 X (DML의 경우에는 무시 / **DDL의 경우에만 영향**)

## 📌 네임드 락

- **네임드 락: 임의의 문자열에 대해 잠금 설정 → GET_LOCK() 함수로 설정**
- 잠금 대상이 데이터베이스 객체가 아닌 **사용자 지정 문자열에 대한 획득/반납(해제)**
- 여러 웹 서버가 하나의 데이터베이스 서버에 접속하는 경우 사용 (**여러 클라이언트가 상호 동기화 처리**)

## 📌 메타데이터 락

- **메타데이터 락: 데이터베이스 객체(테이블, 뷰)의 이름 혹은 구조를 변경할 때 획득하는 락**
- 명시적으로 획득하거나 해제가능한 잠금 X → **자동으로 획득하는 잠금**
- **RENAME TABLE** 명령어로 이름 변경 시, 자동으로 획득

```java
mysql> RENAME TABLE rank TO rank_backup, rank_new TO rank;
```

# 3. InnoDB 스토리지 엔진 잠금

- **InnoDB 스토리지 엔진**: 스토리지 엔진 내부에서 레코드 기반의 잠금 방식 탑재
- 레코드 기반의 잠금 방식 → MyISAM보다 뛰어난 **동시성 처리**를 제공
- 이원화된 잠금 처리 → **InnoDB 스토리지 엔진의 잠금 정보를 MySQL 서버에서 접근**하기 까다로움

## 📌  InnoDB 스토리지 엔진의 잠금

- InnoDB 스토리지 엔진은 **레코드 기반의 잠금 기능을 제공,** 잠금 정보가 **작은 공간으로 관리**
    
    → 레코드 락이 페이지 락 or 테이블 락으로 **레벨업되는 경우는 없음**
    
- 일반 DBMS와 다르게, InnoDB 스토리지 엔진에서는 레코드 락 뿐만 아니라 **레코드와 레코드 사이의 간격을 잠그는 갭락**이 존재

### ✅ 레코드 락

- **레코드 락: 레코드 자체만을 잠그는 락**
- (상용DBMS와 차이는) InnoDB 스토리지 엔진은 레코드 자체가 아니라 **인덱스의 레코드를 잠금**
    - 인덱스가 하나도 없는 테이블이라도 내부적으로 자동 생성된 클러스터 인덱스를 사용하여 설정

### ✅ 갭 락

- **갭 락: 레코드와 인접한 레코드 사이의 간격만을 잠그는 락**
- 레코드와 레코드 사이의 간격에 새로운 레코드가 생성되는 것을 제어
- 넥스트 키 락의 일부로 자주 사용

### ✅ 넥스트 키 락

- **넥스트 키 락: 레코드 락과 갭 락을 합쳐 놓은 형태의 락**
- 바이너리 로그 포맷을 ROW 형태로 바꿔 넥스트 키 락이나 갭 락 줄이는 것이 좋음

### ✅ 자동 증가 락

- MySQL → **자동 증가하는 숫자 값을 추출**하기 위해 **AUTO_INCREMENT 칼럼 속성을 제공**
- AUTO_INCREMENT락 테이블 수준의 잠금을 사용
- **새로운 레코드 저장(INSERT, REPLACE)하는 쿼리**에서만 필요
    - 해당 쿼리 문에서  **AUTO_INCREMENT 값을 가져오는 순간에만 락이 걸림**
- 명시적으로 획득하고 해제하는 방법 X → 아주 짧은 시간 잠금/해제되어 대부분 문제 X
- MySQL 8.0부터 기본값: 시스템 변수 `innodb_autoinc_lock_mode` = 2 (권장)

## 📌 인덱스와 잠금

- InnoDB의 잠금은 레코드를 잠그는 것이 아니라 **인덱스를 잠그는 방식으로 처리**
    - 즉, 변경해야 할 레코드를 찾기 위해 **검색한 인덱스의 레코드를 모두 락**
- MySQL InnoDB에서 **인덱스 설계**가 중요한 이유 →  **인덱스가 없는 경우, 테이블을 전체 스캔하면서 UPDATE 작업을 하기에 테이블에 있는 모든 데이터를 잠금**

.

## 📌 레코드 수준의 잠금 확인 및 해제

- 테이블 잠금에서는 잠금의 대상이 테이블 자체 → 쉽게 문제의 원인이 발견되고 해결
- **레코드 수준의 잠금은 테이블의 레코드 각각에 잠금이 걸림** → 레코드가 자주 사용안되면 **오랜 시간 잠겨진 상태로 남아 있어도 잘 발견 못함**
- MySQL 5.1부터 레코드 잠금과 잠금 대기에 대한 조회 가능 → 쿼리 하나만 실행해보면 잠금, 잠금 대기 확인 가능
- **MySQL 5.1 이후:** information_schema DB의 INNODB_TRX 테이블과 INNODB_LOCKS, INNODB_WAITS 테이블을 통해 확인 가능
- **MySQL 8.0 이후:** information_schema 의 정보들이 점점 제거됨. 그대신 performance_schema의 data_locks와 data_lock_waits 테이블로 대체

# 4. MySQL의 격리 수준

- **트랜잭션의 격리 수준**: **여러 트랜잭션이 동시에 처리될 때 특정 트랜잭션이 다른 트랜잭션에서 변경하거나 조회하는 데이터를 볼 수 있게 허용할지 말지를 결정**
- 종류: **READ UNCOMMITTED, READ COMMITTED, REPETABLE READ, SERIALIZABLE**
    - 뒤로 갈수록 트랜잭션 간의 **데이터 격리(고립)정도가 높아짐. 동시 처리 성능 떨어짐**
    - **MySQL 서버 처리 성능이 떨어지는 것은 X** (SERIALIZABLE이 아니라면 크게 성능 저하 X)

1. **READ UNCOMMITTED ( =**DIRTY READ)
    - **일반적인 데이터베이스에서 잘 사용 X**
2. **READ COMMITTED**
3. **REPEATABLE READ**
4. **SERIALIZABLE**
    - **동시성이 중요한 데이터베이스에서는 거의 사용 X**

- **부정합 문제 (3가지 문제)**
    
    
    |  | DIRTY READ | NON-REPEATABLE READ | PHANTOM READ |
    | --- | --- | --- | --- |
    | READ UNCOMMITTED | 발생 | 발생 | 발생 |
    | READ COMMITTED | 없음 | 발생 | 발생 |
    | REPEATABLE READ | 없음 | 없음 | 발생 (InnoDB는 없음) |
    | SERIALIZABLE | 없음 | 없음 | 없음 |

## 📌 READ UNCOMMITTED

- **각 트랜잭션의 변경 내용이 COMMIT이나 ROLLBACK 여부에 관계없이 다른 트랜잭션에서 보임**
- DIRTY READ가 허용되는 격리 수준
- **DIRTY READ**: 어떤 트랜잭션에서 **처리한 작업이 완료되지 않았는데도** 다른 트랜잭션에서 볼 수 있는 현상

## 📌 READ COMMITTED

- **오라클 DBMS에서 기본으로 사용**되는 격리 수준
- **온라인 서비스에서 가장 많이 선택**
- DIRTY READ 문제 발생 X → **COMMIT이 완료된 데이터만 확인**할 수 있기 때문
- **NON-REPEATABLE READ 문제** 발생 → **REPEATABLE READ가 불가능**
    - 하나의 트랜잭션 내에서 똑같은 SELECT 쿼리를 실행했을 때는 항상 같은 결과를 가져와야함
    - 일반적인 웹 프로그램에서는 크게 문제 X, 하나의 트랜잭션에서 **동일 데이터를 여러 번 읽고 변경하는 작업**이 **금전적인 처리와 연결**된 경우 문제

## 📌 REPEATABLE READ

- **MySQL의 InnoDB 스토리지 엔진에서 기본으로 사용**되는 격리 수준
- **바이너리 로그를 가진 MySQL 서버**에서는 **최소 REPEATABLE READ 이상의 격리 수준을 사용**해야함
- NON-REPEATABLE READ 문제 발생 X
- REPEATABLE READ는 **언두 영역에 백업된 이전 데이터를 이용**해 **동일 트랜잭션 내에서는 동일한 결과**를 보여줄 수 있게 보장
- **PHANTOM READ 문제** 발생 → 다른 트랜잭션에서 수행한 변경 작업에 의해 레코드가 보였다 안보였다하는 현상

## 📌 SERIALIZABLE

- **가장 단순한 격리 수준이면서 동시에 가장 엄격한 격리 수준**
- **동시성 처리 성능이 가장 떨어짐**
- InnoDB 테이블에서 기본적으로 순수한 SELECT 작업은 아무런 레코드 잠금을 설정하지 않고 실행
    
    **→ Non-lockig consistent read (잠금이 필요없는 일관된 읽기)**
    
- 트랜잭션의 격리 수준이 SERIALIZABLE → **읽기 작업도 공유 잠금(읽기 잠금)을 획득**해야만 하며, 동시에 다른 트랜잭션은 해당 레코드를 변경 X
